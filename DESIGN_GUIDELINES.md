## デザイン原則

### センスは知識からはじまる

デザインのセンスは才能やセンスではなく、**知識を蓄積することで向上する能力**です。
このガイドラインに従うことで、一貫性のあるデザインを実現できます。

## カラーシステム

### 3色の配色ルール

**全体で3色に統一**します。色を追加する前に「本当に必要か」を検討してください。

#### カラーパレット

| 役割 | カラー | 使用比率 | 用途 |
|------|--------|---------|------|
| ベースカラー | `oklch(0.985 0.001 0)` (ライト) | 70% | 背景、大面積 |
| メインカラー | `oklch(0.530 0.130 184)` (ティールブルー) | 25% | プライマリアクション、重要要素 |
| セカンダリカラー | `oklch(0.610 0.160 184)` (アクセント) | 5% | 強調、アクセント |

### 色の組み合わせ

- ✅ **やること**：目立たせたい要素だけに色を使う
- ❌ **避けること**：「あれもこれも大事だから」と多くの色を使う

### コントラスト比（視認性）

WCAG 2.0基準に従います：

- **小さい文字（14px未満）**：4.5以上の比率を推奨
- **大きい文字（18pt以上）**：3.0以上の比率を推奨

```
背景：#FFFFFF
良い例：#333333（濃いグレー、コントラスト 12.6）
悪い例：#FFFF00（黄色、コントラスト 1.0）
```

## タイポグラフィ

### フォント

**フォントは1つだけ使用**します。

- **フォント選択**：`Geist`（Tailwind + Next.jsのデフォルト）
- **代替フォント**：`Geist Fallback`

### ジャンプ率

フォントサイズの階層には一貫した比率を使用します。
**ジャンプ率：1.25（Major Third）**

```
大見出し：    28px
↓ ×1.25
小見出し：    24px
↓ ×1.25
本文：        16px
↓ ×1.25
キャプション：12px
```

### 行の高さ

テキストは読みやすく、目に優しい行の高さにします。

- **行の高さ**：1.4～1.6（Tailwindでは `leading-relaxed` など）
- **一般的**：1.5（標準的なバランス）

### テキスト比率：漢字とひらがな

日本語の可読性を高めるため、次の比率を目安にします：

- **ひらがな**：70%（親しみやすく読みやすい）
- **漢字**：30%（意味を素早く把握）

```
❌ 悪い例：「保存完了後に処理を実行してください」
✅ 良い例：「保存が終わったら、処理を実行します」
```

### 中央揃えの使用

- ❌ **避ける**：情報の関連性が不明確になる
- ✅ **推奨**：左揃えで一体感を出す
- ⚠️ **例外**：前後の関係を意図的に断ち切りたい場合のみ（例：タイトル）

## レイアウト

### 4つの基本原則

#### 1. 近接（Proximity）

関連する項目を近づけてグループ化します。
「遠いものより近くのものが関連する」と認識されます。

```
余白の大きさ = 関連性の弱さ

８px（強い関連）
１６px（中程度）
２４px（別グループ）
３２px（セクション間）
```

#### 2. 整列（Alignment）

すべての要素を見えない線で整列させます。
これにより一貫性と一体感が生まれます。

- **左揃え**：情報を結びつける直線が見えて一体感がある ✅
- **中央揃え**：結びつける線が見えず、グループ感が薄い ❌

#### 3. 反復（Repetition）

デザイン上の特徴を全体を通して繰り返します。
全体の一貫性を生み出します。

```
例：
- ヘッダ・フッターを統一
- スペーシング比率を統一
- 色の使用を統一
- ボタンスタイルを統一
```

#### 4. コントラスト（Contrast）

2つの要素が異なることを、サイズや太さで対比させます。
階層構造や情報の構造化のために使用します。

```
❌ 悪い例：すべて同じサイズ、重さ
✅ 良い例：
  - 大見出し：28px / 600weight
  - 本文：16px / 400weight
  → 視線の流れが明確になる
```

## UXライティング

### 3つの核となる原則

#### 1. ユーザーを主体者にする

システムではなく、**ユーザーが何ができるか**を示します。

```
❌ システム主体：「顔認証ができます」
✅ ユーザー主体：「私は顔認証で入室ができます」
```

#### 2. 話し言葉で親切に

書き言葉ではなく、対面で会話するような語調を使います。

```
❌ 書き言葉：「ファイル削除を実行してください」
✅ 話し言葉：「ファイルを削除します」

❌ 漢字が多い：「機能を追加する」（わかりにくい）
✅ バランスが良い：「機能を追加できます」（わかりやすい）
```

#### 3. ボタンラベルは動詞で明確に

「はい/いいえ」ではなく、押したら何が起こるかを動詞で表示します。

```
❌ あいまい：
  「削除してもよろしいですか？」
  [ はい ]  [ いいえ ]
  
✅ 明確：
  「本当に削除しますか？」
  [ 削除する ]  [ キャンセル ]
```

### UXライティング チェックリスト

各機能を実装するときに確認してください：

- [ ] ユーザーを主体者にしているか？（「～できます」「～します」）
- [ ] ユーザーを責めていないか？（エラーメッセージで「してください」を避けている）
- [ ] ボタンラベルは動詞で終わっているか？（「ログインする」「削除する」）
- [ ] 説明文を追加して親切にしているか？
- [ ] 話し言葉で書かれているか？（漢字3割：ひらがな7割）
- [ ] 専門用語を避けているか？
- [ ] コントラスト比は十分か？

### エラーメッセージのルール

ユーザーを責めず、親切に対処方法を説明します。

```
❌ ユーザーを責める：
  「エラーが発生しました」
  
✅ 親切に説明：
  「申し訳ありません。メールアドレスまたはパスワードが正しくありません。」
  「もう一度確認してからお試しください。」
```

### ボタンテキストの統一

文章とボタンで**同じ動詞を使う**ことが大切です。

```
❌ 動詞が異なる：
  「商品をオーダーしますか？」
  [ 中止する ]  [ 注文する ]  ← 「注文」と「オーダー」

✅ 同じ動詞：
  「商品を注文しますか？」
  [ 中止する ]  [ 注文する ]
```

## 実装ガイド

### ファイル参照

このアプリケーション内の実装例：

- **カラーシステム**：`app/globals.css`（OKLch色空間で定義）
- **UXライティング**：`lib/ux-strings.ts`（ユーザーフレンドリーな文言）
- **デザインシステム**：`lib/design-system.ts`（デザイン原則の実装）

### 新しい機能を追加するときの流れ

1. **UIをデザインする**
   - 3色に統一しているか確認
   - レイアウトの4原則に従っているか確認

2. **テキストを書く**
   - `lib/ux-strings.ts` から適切な文言を選択
   - ユーザーを主体者にしているか確認
   - 話し言葉か確認

3. **コンポーネントを実装する**
   - Tailwind CSSで色定数を使用（`bg-primary` など）
   - ボタンラベルに動詞を使用
   - 説明文を含める

## 質問？問題が出た？

このガイドラインに従っていても問題が生じた場合は、
まずこのドキュメントを読み直してみてください。
ほとんどの問題は、3つの基本原則を見直すことで解決します。

**大切なのは、ルールを厳密に守ることではなく、
ユーザーの視点で「わかりやすい」「使いやすい」デザインを作ることです。**
